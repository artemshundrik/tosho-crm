## 1. Мета модуля

Дати можливість з CRM **одним натиском** відправляти вихідні документи (договори, додатки, СП, ВН, акти, рахунки) у «Вчасно» контрагенту — без ручного перекидання файлів — і **бачити в CRM статуси + сповіщення** про вхідні документи та підписи контрагентів.

Не входить у скоуп: сам КЕП-підпис лишається ручним (ставить власник ключа / бухгалтер у Вчасно). API лише завантажує, надсилає, відстежує.

## 2. Базові поняття

### 2.1. Юрособи ↔ кабінети ↔ токени

Три **окремі кабінети** Вчасно, кожен зі своїм API-токеном. Документ іде з-під тієї юрособи, що його виставляє (визначається з фінмодуля, менеджер нічого не обирає вручну):

| Юрособа | Кабінет | Токен (env) | Статус |
|---|---|---|---|
| ФОП Хом'яков В.О. | окремий | `VCHASNO_TOKEN_FOP_VO` | ✅ токен є, перевірений |
| ФОП Хом'яков Р.О. | окремий | `VCHASNO_TOKEN_FOP_RO` | очікуємо токен |
| ТОВ «АВАНПРІНТ» | окремий | `VCHASNO_TOKEN_AVANPRINT` | очікуємо токен |

Прив'язка до існуючих `tosho.finance_legal_entities` (фінмодуль уже має юрособи) — додати поле-ключ (`vchasno_company_key`), за яким бекенд бере потрібний токен з env. Токени — **тільки в серверних секретах** (Netlify env / Supabase), ніколи не на фронті й не в коді.

### 2.2. Типи документів (мапа на `category` Вчасно)

| Наш документ | `category` |
|---|---|
| Рахунок | 2 |
| Видаткова накладна (ВН) | 5 |
| Специфікація (СП) | 9 |
| Договір | 3 |
| Додаток / додаткова угода | 14 / 4 |
| Акт наданих послуг | 1 |
| Акт звіряння | 16 |

### 2.3. Три потоки

1. **Вихідний** — CRM генерує PDF → `POST /documents` (завантаження під потрібним токеном) → `POST /documents/{id}/flow` (надсилання контрагенту за ЄДРПОУ/email). Підпис ставить людина у Вчасно.
2. **Статуси** — періодичне опитування `POST /documents/statuses` (по нашому `vendor_id`) або `GET /documents?has_changed=1` → оновлюємо статус у CRM, сповіщаємо коли контрагент підписав / відхилив.
3. **Вхідний** — `GET /incoming-documents?processed=false` → новий вхідний → сповіщення → `POST /documents/mark-as-processed`.

### 2.4. Права: завантаження vs надсилання

Дві різні дії з різними правами:
- **Завантаження** (`POST /documents`, чернетка у Вчасно) — доступне ширшому колу (команда, що готує документи).
- **Надсилання контрагенту** (`POST /documents/{id}/flow`) — лише **уповноважена особа** (директор або головбух). Не всі підряд.

Реалізуємо як два окремі дозволи (напр. `vchasno.upload` / `vchasno.send`); конкретних осіб призначають у налаштуваннях доступів (уточнять пізніше).

### 2.5. Документи Фази 1

Першими автоматизуємо: **видаткова накладна (5), рахунок (2), додаток (14), договір (3)**. Специфікація та акти — пізніша фаза.

## 3. API-контракт (перевірено)

- Хост: `https://edo.vchasno.ua`, база `/api/v2/` (актуальна; `/api/v1` застаріла — без `cursor`).
- Авторизація: заголовок `Authorization: <token>` у **кожному** запиті. Токен = зв'язка користувач+компанія.
- Ліміти завантаження: ≤500 файлів за раз, ≤100 Мб сумарно, ≤15 Мб на файл. Можна `.zip` до 500 PDF.

### 3.1. Потрібні ендпоінти

| Завдання | Метод |
|---|---|
| Перелік + статуси вихідних | `GET /api/v2/documents` (фільтри: `vendor_id`, `has_changed`, `recipient_edrpou`, дати, `ids`) |
| Статуси поштучно по ID | `POST /api/v2/documents/statuses` |
| Завантажити документ | `POST /api/v2/documents` |
| Надіслати контрагенту | `POST /api/v2/documents/{id}/flow` |
| Додати підписантів | `POST /api/v2/documents/{id}/signers` |
| Вхідні документи | `GET /api/v2/incoming-documents` (фільтр `processed`) |
| Позначити вхідний обробленим | `POST /api/v2/documents/mark-as-processed` |
| Оригінал / .p7s з підписами / ASIC | `GET /api/v2/documents/{id}/original` · `/p7s` · `/asic` |
| Друкована PDF-візуалізація | `GET /api/v2/documents/{id}/pdf/print` |
| Підпис КЕП через API (опційно, Фаза 4) | сесія підпису (вимагає пароль ключа / додаток Вчасно.КЕП) |

### 3.2. Формат завантаження

Метадані передаються **або іменем файлу**, або параметрами запиту (тоді ім'я довільне):

```
<edrpou_owner>_<edrpou_recipient>_<date YYYYMMDD>_<title>_<number>_<email_recipient>_<vendor_id>_<amount копійки>.pdf
```

- `vendor_id` = наш `crm_doc_id` → за ним синхронізуємо статуси назад у CRM.
- Якщо передано `email_recipient` → документ одразу в статусі **7001 «Готовий до підпису та надсилання»**; без email → **7000 «Завантажений»**.
- `expected_owner_signatures` / `expected_recipient_signatures` — скільки підписів треба.

### 3.3. Ключові статуси

`7000` завантажений · `7001` готовий до підпису/надсилання · `7006`/`7008` завершено (підписано) · `7011` анульовано. Повна мапа — в офіційній інструкції API.

## 4. Архітектура в нашому стеку

```
CRM (React) ──► Netlify Function (proxy, тримає 3 токени) ──► edo.vchasno.ua/api/v2
       ▲                          │
       │                          ▼
   нотифікації            tosho.vchasno_documents (Supabase)
   (CRM/TG/push)                  ▲
                                  │
                   pg_cron (кожні N хв) ──► Netlify Function (poll статусів + вхідних)
```

- **Netlify Function-проксі** — усі виклики Вчасно йдуть через сервер (як інші привілейовані операції). Фронт ніколи не бачить токенів.
- **Генерація PDF** — *ключовий технічний момент*: фінмодуль зараз робить документи через HTML→`window.print()` (браузерний друк), а Вчасно приймає **готовий файл PDF**. Треба програмний шлях: рендер того ж HTML у PDF (headless-рендер на сервері або PDF-бібліотека на клієнті → байти у функцію). Це винести в Фазу 1.
- **Опитування** — переюзати інфру нагадувань (`pg_cron` + Netlify Function + дедуп) замість мертвого Netlify-скедулера. Вебхуків у Вчасно публічно немає → тільки polling.
- **Сповіщення** — переюзати наявне: in-app CRM + Telegram-бот + пуш (`docs/push-notifications.md`). Слати всі три канали.

## 5. Дані (пропозиція, DRAFT)

Нова таблиця `tosho.vchasno_documents` (RLS team-scoped, як решта):

| Поле | Тип | Призначення |
|---|---|---|
| `id` | uuid pk | |
| `legal_entity_id` | uuid → finance_legal_entities | з-під кого слали (→ який токен) |
| `crm_doc_type` | text | рахунок/вн/сп/акт/договір/додаток |
| `crm_doc_id` | uuid | наш документ (напр. `finance_invoices.id`) = `vendor_id` |
| `order_id` / `quote_id` | uuid null | зв'язок із замовленням/КП |
| `direction` | text | outgoing / incoming |
| `vchasno_document_id` | text | ID у Вчасно |
| `vchasno_category` | int | мапа типу |
| `recipient_edrpou` | text | ЄДРПОУ контрагента |
| `status_code` | int | останній статус (7000…7011) |
| `status_label` | text | людський опис |
| `sent_at` / `signed_at` / `finished_at` | timestamptz | віхи |
| `last_synced_at` | timestamptz | коли востаннє опитували |
| `raw` | jsonb | остання відповідь API (дебаг) |
| + аудит (`created_by`, `team_id`, `created_at`) | | |

На `finance_legal_entities` — додати `vchasno_company_key` (текстовий ключ → env-токен) та переконатися, що є ЄДРПОУ/ІПН (потрібен як `edrpou_owner`).

**Картка клієнта — нова вкладка «Бухгалтер»** з полями: `accountant_email` (email підписанта/бухгалтера, куди слати документ) і `accountant_edrpou` (ЄДРПОУ/ІПН контрагента = `recipient_edrpou`). Підписант з боку контрагента — зазвичай бухгалтер або керівник. `accountant_email` → `email_recipient` у Вчасно; якщо порожнє — фолбек на основний email клієнта. Поточний основний email не завжди той, на який треба слати документи.

## 6. Безпека

- 3 токени — лише в серверних секретах (Netlify/Supabase env). Жодного на фронті, жодного в репозиторії.
- **НЕ вмикати** у Вчасно «доступ тільки з певних IP» (Налаштування безпеки) — Netlify Functions не мають статичної вихідної IP, обмеження зламає API-виклики (стосується і сесії підпису КЕП).
- КЕП-підпис лишається ручним — приватний ключ у системі не зберігаємо.
- Перед «готово» — обовʼязково `/security-review` (Netlify Functions, привілейовані записи, секрети) і `/review` (зміни в гігантських сторінках замовлень/фінансів).

## 7. Етапність

- **Фаза 1 — вихідний потік, 1 юрособа (ФОП В.О.).** Типи: ВН, рахунок, додаток, договір. PDF-генерація → `POST /documents` (завантаження — ширший доступ) → кнопка **«Відправити у Вчасно»** лише для уповноваженої особи → `POST /flow`. Запис у `vchasno_documents`.
- **Фаза 2 — статуси + вхідні + сполучення.** pg_cron-опитування статусів і `incoming-documents`; сповіщення CRM/TG/пуш; бейдж статусу на картці.
- **Фаза 3 — масштабування на 3 юрособи.** Підключити токени ФОП Р.О. і ТОВ «АВАНПРІНТ», маршрутизація токена за юрособою.
- **Фаза 4 (опційно) — підпис КЕП через API.** Сесія підпису у Вчасно.КЕП, якщо захочуть підписувати прямо з CRM (вимагає пароль ключа / додаток).
