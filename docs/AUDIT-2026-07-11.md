# Project Audit — tosho-crm (2026-07-11)

Read-only audit across four dimensions (security/authz, frontend health, performance/data,
TypeScript/tooling), run by four parallel agents grounded in `AGENTS.md` + `docs/CODEX_*`.
Security findings marked **PROVEN** were verified at runtime via `psql` + REST (read-only),
not eyeballed. This report doubles as the fix backlog and the skeleton for per-module docs.

**Verification note:** the P0 leak below was proven end-to-end — an anonymous REST call with
only the public `anon` key (no login token) returned `HTTP 200` with real rows, and
`set role anon; select count(*)` returned live row counts. Method is in `docs/SECURITY.md`
§"Verify by simulating the role".

---

## 🔴 P0 — CRITICAL: Unauthenticated data leak via anon-granted views

**PROVEN.** Four Postgres views are `GRANT SELECT ... TO anon` **and** created without
`security_invoker`, so they run with owner (`postgres`) privileges and bypass base-table RLS.
`anon` = the public key shipped in the JS bundle = anyone on the internet.

| View | Rows visible to anon | Data exposed |
|---|---|---|
| `tosho.v_quotes_list` | **234** | every quote: customer names, totals, deadlines, logos |
| `tosho.memberships_view` | **18** | employee memberships / access roles |
| `tosho.workspace_member_directory` | **18** | employee directory (PII) |
| `public.team_members_view` | **20** | team members (PII) |

Proof: `GET /rest/v1/v_quotes_list?select=id&limit=3` with only the `anon` apikey and **no
Authorization header** → `HTTP 200`, 3 rows. This is the exact anti-pattern `docs/SECURITY.md`
§1 warns about.

**Fix (safe, reversible):** `REVOKE SELECT ON <view> FROM anon` for all four. The app reads
these only as `authenticated` (logged-in frontend) or `service_role` (Netlify functions) —
confirmed by grep; no anonymous read path exists. Ready-to-run: `scripts/fix-anon-view-leak.sql`.
Hardening follow-up: recreate the views `WITH (security_invoker = on)` after per-view RLS
review, so a future accidental `anon` grant can't leak.

### Related (lower severity): 11 RLS-off base tables with anon grant

`design_task_number_counters`, `_healthcheck`, `catalog_{kinds,methods,model_methods,models,
price_tiers,print_positions,types}`, `name_declensions`, `quote_counters` have RLS off + anon
grant. These are **catalog/counter/utility** data — low confidentiality, but they violate
deny-by-default and should be tidied (revoke anon or enable RLS). **Good news:** the sensitive
base tables (`customers`, `quotes`, `finance_*`, `payroll_entries`, `leads`, `contractors`)
are **not** in this set — they have RLS enabled, which filters anon to nothing.

## 🟠 P1 — HIGH

### Security

1. **Six cron functions publicly invokable, no shared-secret gate.** `activity-log-retention.ts`,
   `probation-reminders.ts`, `customer-lead-reminders.ts`, `contractor-reminders.ts`,
   `quote-deadline-reminders.ts`, `team-events-reminders.ts` run with the service-role key and
   check no secret. `scripts/reminders-cron.sql:27-57` sends no auth header (TODO at :85-86).
   Impact: anyone can drive notification spam (in-app + Telegram + push) on a loop, and trigger
   `archive_activity_log_all` (audit-trail archival/DoS). **Known-open from the prior audit; still open.**
   Fix requires the secret check **and** updating the pg_cron `net.http_post` calls to send it — coordinate both sides.
2. **`attachment-preview-generate.ts:76-89` — IDOR via service role.** Verifies JWT but then
   downloads a body-supplied `(bucket, storagePath)` with the service-role client, no ownership
   check. Any authenticated employee can force a service-role read of arbitrary bucket paths.
   Contrast the correct pattern in `dropbox-export.ts:91-114`.
3. **`notify-users.ts:89-100` — authenticated but not authorized.** Delivers notifications to
   arbitrary body-supplied `userIds` with attacker-chosen `title`/`body`/`href`. Any logged-in
   user can push arbitrary in-app/Telegram/push notifications to anyone (internal phishing via `href`).
4. **`catalog-image-import.ts:156,176` — SSRF + arbitrary storage write.** Fetches a body-supplied
   `sourceUrl` server-side with no domain allowlist (sibling `catalog-avanprint-import.ts:526`
   validates its URL; this one doesn't), and writes to body-supplied bucket/path via service role.

### Performance / data access

5. **`loadDerivedOrders` (`src/features/orders/orderRecords.ts`) is the app's hot path and is a
   double-N+1 + unbounded full-history load with no cache** — called from ~10 sites (production
   list, order details, finances, customer quick view, Cmd+K palette).
   - N+1: one `quote_item_runs` query per quote (`orderRecords.ts:650,1027` → `getQuoteRuns`
     `toshoApi.ts:1157`). Fix: batch with a single `.in("quote_id", quoteIds)` — the sibling
     `listQuoteItemsForQuotes` (`toshoApi.ts:1961`) is the template.
   - Unbounded: `listStoredOrders` (`:528`) and `listQuotes({status:"approved"})` (`:582`) have no
     `.limit`; grow forever, always loaded in full.
   - "Load everything to display one": `OrdersProductionDetailsPage.tsx:1182,1201` build the whole
     dataset to `.find(id)` — twice.
   - Cmd+K (`CommandPalette.tsx:649`) rebuilds the entire derivation on 180ms-debounced keystrokes.
6. **Indexes unverified on hot paths** (tables created outside tracked SQL): confirm
   `quote_item_runs(quote_id)`, `activity_log(team_id, action, created_at)`,
   `quotes(team_id, status, updated_at)` exist — a missing `quote_item_runs(quote_id)` turns #5
   into N sequential scans.

### Frontend

7. **Four god-components with no memo boundaries → whole-page re-render storms.**
   `QuoteDetailsPage.tsx` (142 `useState`, ~9,900 lines), `DesignTaskPage.tsx` (128, ~12,000),
   `QuotesPage.tsx` (109), `DesignPage.tsx` (86). `React.memo` appears in only 2 files repo-wide.
   Every keystroke in the comment composer (`QuoteDetailsPage.tsx:786`→input `:7984`) re-renders
   the entire tree; React Compiler is **not** in the build, so nothing bails this out.
   Fix: extract high-frequency-input subtrees (comment/brief/notes editors, search) into
   `React.memo` children with stable `useCallback`.
8. **The four mega-pages need decomposition along clear state clusters** (edit-dialog, runs,
   comments, brief, notes…) into colocated hooks/components (`useQuoteRuns`, `useQuoteComments`,
   `<QuoteBriefEditor>`, `<EditQuoteDialog>`). This also enables #7's memo boundaries.

### TypeScript / tooling

9. **Entire Supabase/DB surface is untyped `any`.** `src/lib/supabaseClient.ts:5-7,42` declares
   the client `SupabaseClient<any,...>`; no generated `Database` type exists. ~734 `.from()` and
   ~274 writes are unchecked — including money/pricing (`quote_item_runs`). `strict: true` gives
   false confidence. Fix: `supabase gen types typescript` and parametrize the client.
10. **`npm run lint` is RED on a clean checkout — 69 errors, 2 warnings.** CLAUDE.md names lint as
    default verification, but it doesn't pass: 49 `no-unused-vars` (dead code), 16
    `exhaustive-deps`, 3 `preserve-manual-memoization` (genuine memo/stale-closure hazards).
    Fix: clear to zero, then keep CI red on regressions. (`npx tsc --noEmit` is **clean**.)
11. **30 Netlify functions are never type-checked.** `npm run build` = `tsc -b && vite build`;
    root `tsconfig.json` has `include: ["src"]` only; functions bundle via esbuild (no check). The
    security-sensitive server tier has zero compile-time safety. Fix: add `tsconfig.functions.json`
    + `tsc --noEmit -p` in CI.
12. **Unchecked Supabase writes swallow rejections.** `QuoteDetailsPage.tsx:1293` deletes the
    pricing source of truth (`quote_item_runs`) with no `error` capture, then proceeds as if it
    worked — the "looks applied but was rejected" bug class (siblings: `AppLayout.tsx:1254`,
    `NotificationsPage.tsx:803`, `activityLogger.ts:45`). Fix: check `error` on every write or
    wrap writes in a throwing helper.

- **No `src/lib/format.ts`** → tiny helpers reimplemented: `getInitials` ×15 (canonical
  `getInitialsFromName` already at `src/lib/userName.ts:84`), `formatDate` ×5 identical, 21
  scattered `Intl.NumberFormat` money formatters with divergent rounding, `formatFileSize` ×3,
  `escapeHtml` ×5. Fix: one `src/lib/format.ts`; codemod the copies.
- **`DesignTask` type + cache sanitizers duplicated** across `DesignPage.tsx:103/669/681` and
  `DesignTaskPage.tsx:189/793/805` — the sanitizer's own comment warns it "poisons writes" if
  metadata is stripped; two drifting copies of a correctness trap. Fix: `src/lib/designTaskCache.ts`.
- **Quote status config duplicated** (`statusLabels`/`normalizeStatus` byte-identical in
  `features/quotes/quotes-page/config.ts` and `quote-details/config.tsx`). Fix: shared `status.ts`.
- **Print/export HTML hand-rolled ×4-5 pages** (`OrdersProductionPage`, `OrdersProductionDetailsPage`,
  `QuotesPage`, `finances/documentHtml.ts`). Fix: `src/lib/printDocument.ts`.
- **Per-page session-cache read/write reimplemented ×6** despite `src/lib/sessionCache.ts`. Fix:
  generic `readSessionCache<T>`/`writeSessionCache`.
- **~33 inline role checks** (`=== 'owner'`, `access_role ===`) bypass `src/lib/permissions.ts`.
  Fix: route gating through named permission predicates.
- **Overview landing fans out ~8 count round-trips** (`OverviewPage.tsx:741,754,759`). Fix: one
  grouped aggregate read.
- **No test infrastructure at all** — no vitest/jest, no `*.test.*`. Highest-value untested logic:
  `computeRunSalePricing`/`mergeQuoteRunsWithExisting` (`src/lib/quoteRuns.ts`), order derivation,
  permissions — all pure functions, ideal unit targets. Fix: add vitest, start with pricing.
- **Orphaned strict tsconfigs** — root `tsconfig.json` has no `references`, so
  `tsconfig.app.json`'s `noUnusedLocals`/`noUnusedParameters`/etc. never apply. Fix: restore refs.

## Follow-ups surfaced during module documentation (2026-07-12, UNVERIFIED)

Observations flagged by the per-module doc pass (`docs/modules/`). These are **not confirmed
findings** — verify before acting.

- **Module-access may be UI-only on some routes.** `marketing`, `design`, and `catalog` routes
  appear to lack a route-level `ModuleRouteGate`; for `catalog` and `marketing` the table RLS is
  `is_team_member(team_id)` only (not role-gated), so the owner/SEO/marketer module restriction
  could be **frontend-only** — any authenticated team member might read/write via the REST API.
  Scope is within-team (not anonymous), so low-medium. **Verify** the RLS policies + whether nav
  gating is the only enforcement.
- **Sample-stock ledger is bypassable.** The item-edit Sheet writes `quantity_on_hand`/
  `reserved_quantity` via a plain `.update()`, bypassing the `adjust_sample_stock_item` ledger
  RPC — so `sample_stock_movements` is not a complete audit history. Also `currency` is hardcoded
  to `UAH` on save. The movement ledger has no UI at all.
- **Team-HR profile dual-write.** Profile fields are written to both `tosho.team_member_profiles`
  and auth `user_metadata` (`create-workspace-invite.ts` `update_member_profile` mode) — authority
  per field is unclear; a rejected write could look partially applied.
- **DB_MAP gaps.** `marketing_visuals`, `contractors`, `contract_revisions` are absent from
  `docs/DB_MAP.md`; `backup_runs` / `runtime_errors` DDL is not in tracked `scripts/*.sql` (DB-only).
- **Possibly-dead/unwired code.** `tosho.crm_contacts` has zero `src/` references (contacts live as
  jsonb); `runtime_errors` and `netlify/functions/admin-attachment-audit.ts` are not surfaced in the
  Admin Observability UI (parallel/legacy paths).
