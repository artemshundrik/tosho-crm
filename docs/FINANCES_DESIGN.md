>
> Поточний стан коду:
> - Розділ `finance` заведено в навігацію + систему доступів (`module_access.finance`, дефолт `false`, видимий для `owner`/`seo`).
> - **Фаза 1 розпочата.** Схема БД застосована: `scripts/finances-schema.sql` → `tosho.finance_legal_entities`, `finance_accounts`, `finance_order_meta`, `finance_invoices`, `finance_payments` (RLS team-scoped, тригери, smoke-тест пройдено).
> - `src/features/finances/` — типи + CRUD-API + компоненти.
> - **Усі 11 розділів робочі** (крім банк-інтеграції — її користувач підключить пізніше):
>   1. Дашборд — надходження по контурах/касах + дебіторка
>   2. Реєстр продажів — Рахунки (+ генерація рахунку PDF, класифікація товари/послуги → `finance_order_meta`) + Оплати
>   3. Витрати — сталі (закріплені) + змінні; розподіл 1 витрати на N замовлень
>   4. Виплати команді — переюзує `payroll_entries` + оверлей `finance_payout_meta` (юрособа/статус)
>   5. Податки — ПДВ/ЄП/ЄСВ/ВЗ (`finance_taxes`)
>   6. Платіжний календар — несплачені податки + невиплачені зарплати по терміновості
>   7. Каси та рахунки — баланс = надходження − витрати по касі
>   8. Маржа — факт = отримано − розподілені витрати (лише топ-ролі)
>   9. Звірки — генерація акту звірки (рахунки−оплати=сальдо) у PDF + Excel
>   10. Звіти — отримано/витрати/прибуток/дебіторка/податки, фільтр період
>   11. Налаштування — Юрособи / Каси / Статті витрат / Реквізити (швидке копіювання)
> - Схеми застосовані: `finances-schema.sql`, `finances-expenses-schema.sql`, `finances-payout-meta-schema.sql`, `finances-taxes-schema.sql`.
> - Документи генеруються як HTML→popup→`window.print()` (`src/features/finances/documentHtml.ts`); Excel — HTML-таблиця (.xls).
> - **Лишилось (опційно):** банк-інтеграція (Райфайзен+Mono+CSV — користувач сам); установчі-документи file upload; планова маржа з прорахунку; архів згенерованих звірок.

## 1. Мета модуля

Дати керівництву (CEO + головбух) відповіді на питання:

- Скільки заробили / скільки витратили — по кожному контуру (ТОВ / ФОП1 / ФОП2 / фіз / готівка / крипта)
- Хто винен гроші нам (дебіторка) і кому винні ми (кредиторка)
- Що треба оплатити сьогодні/цього тижня (вихідні платежі, податки, зарплати)
- Яке замовлення прибуткове, а яке з'їло маржу (план vs факт по кожному замовленню)
- Скільки замовлень за місяць пройшло через кожен канал (ПДВ / ФОП / готівка / крипта)

**Метод обліку — касовий.** Прибуток визнаємо коли гроші реально зайшли, не коли виставлено рахунок. Прорахунок грошей не отримує — тільки готове замовлення.

**Старт — з чистого аркуша з 01.06.2026.** Історію за 2024–2025 не мігруємо. Потрібна можливість вручну ввести «перші 15 днів» / вхідні залишки.

## 2. Базові поняття

### 2.1. Контури / канали обліку (спина всього)

Кожне замовлення й кожна оплата йдуть через один із каналів. Канал = пара (юрособа + тип розрахунку):

| Канал | Юрособа | Тип | ПДВ | Документи | Видимість |
|---|---|---|---|---|---|
| ТОВ з ПДВ | ТОВ | банк | так | повні (рахунок+видаткова/акт з ПДВ) | усі фін-ролі |
| ФОП №1 | ФОП №1 | банк/IBAN | ні | рахунок без ПДВ | усі фін-ролі |
| ФОП №2 | ФОП №2 | банк/IBAN | ні | рахунок без ПДВ | усі фін-ролі |
| Фізособа | — | особиста картка | ні | немає | **тільки топ-ролі** |
| Готівка | (каса) | готівка | ні | немає | **тільки топ-ролі** |
| Крипта | (гаманець) | крипта | ні | немає | **тільки топ-ролі** |

Замовлення можна фільтрувати по каналу → бачимо за місяць розбивку.

> **Чутливість.** Канали «фізособа / готівка / крипта» — це позаофіційний дохід. У офіційні податкові звіти (ПДВ, ФОП) не потрапляють. Доступ — лише топ-ролі. Враховуються в управлінському прибутку/маржі на дашборді, але окремою лінією.

### 2.2. Каси / гаманці (`finance_accounts`)

Кожен канал має «гаманець» зі своїм балансом:
- ТОВ-банк (Райфайзен — основний рахунок)
- ТОВ/ФОП-Mono (Універсал Банк)
- ФОП №1 IBAN, ФОП №2 IBAN
- Фізкартка
- Каса (готівка, UAH)
- Крипто-гаманець (валюта + курс на момент надходження)

Баланс гаманця = сума надходжень − сума видатків по ньому.

### 2.3. Замовлення — типи

- **Товари**: мерч, брендування, широкоформат тощо → документ реалізації = **видаткова накладна**
- **Послуги**: нанесення, вишивка, перевезення → документ реалізації = **акт виконаних робіт**

Замовлення несе: `order_type` (goods/services), `payment_channel`, `legal_entity_id`.

### 2.4. Витрати — дві групи

- **Сталі (постійні)** — щомісячні, закріплені вгорі: оренда, комуналка, прибирання, смаколики, бензин, інтернет, BAS, Tucha, оплата CRM тощо.
- **Змінні (під замовлення)** — закупівля матеріалів, виробничі витрати. Можуть **ділитись на кілька замовлень** (купили опт на 2-3 замовлення → бухгалтер вручну розподіляє пропорцію).

### 2.5. Податки

- **ТОВ**: ПДВ (окрема стаття витрат). Податкова рахує ПДВ по першій події (не касовий) — облік ПДВ ведемо окремо від управлінського прибутку.
- **ФОП 3 група**: ЄП 5% + ЄСВ + **ВЗ (військовий збір) 1%** — окрема таблиця.

## 3. Структура розділів (навігація всередині «Фінанси»)

> CEO підтвердив об'єднання Рахунків+Оплат+Документів реалізації в один розділ, та розбиття Налаштувань на 4. Нижче — фінальна пропозиція. Назви/порядок — на узгодження.

1. **Дашборд** — фінансова картина по контурах: заробили / витратили / дебіторка / кредиторка / прибуток / маржа; «що оплатити сьогодні»; баланси кас.
2. **Реєстр продажів** *(робоча назва; CEO пропонував «Платіжний реєстр» — але щоб не плутати з «Платіжним календарем», лишаю «Реєстр продажів»)* — рахунки клієнтам + оплати від них + документи реалізації (видаткові/акти). Рядок = рахунок, у розгортці — оплати й документи. Фільтри: канал / тип / статус оплати / контрагент.
3. **Витрати** — дві групи: сталі (закріплені) + змінні (з алокацією на замовлення). Постачальники тут же.
4. **Виплати команді** — зарплати / ФОП-договори, місячні акти, статус виплат.
5. **Податки** — ПДВ (ТОВ) + ФОП (ЄП 5% + ЄСВ + ВЗ 1%), окремими таблицями.
6. **Платіжний календар** — що / кому / коли платити: вихідні платежі + податки + зарплати, великий календар.
7. **Каси та рахунки** — баланси по всіх гаманцях/контурах. (Може бути панеллю на Дашборді + детальний вид.)
8. **Маржа** — план vs факт по кожному замовленню (рахунок − закупки − виробничі витрати − логістика).
9. **Звірки** — генерація актів звірки з даних (рахунки − оплати = залишок), експорт PDF+Excel, архів зі снепшотами + надсилання контрагенту.
10. **Звіти** — продажі, дебіторка, кредиторка, прибуток, ПДВ, розбивки ФОП/ТОВ/канал.
11. **Налаштування** (4 під-розділи):
    - Юрособи (ТОВ + ФОПи; ставки ПДВ, групи)
    - Статті витрат
    - Шаблони документів
    - Установчі документи + реквізити (вивантаження по одному / архівом / надсилання в Telegram/Viber/WhatsApp/email + швидке копіювання реквізитів)

## 4. Документи

- **Генеруємо** з даних системи як готові PDF + Excel: рахунки, видаткові, акти, звірки.
- Виводимо списком з пошуком по фільтру: номер документа / дата / Замовник|Підрядник|Постачальник.
- **Зовнішній підпис**: для контрагентів на ЕДО документи підписуються в Медок/Вчасно (поза CRM). CRM — джерело даних і генератор чернеток; у документа є статус `edm_status` (none / signed_external).

## 5. Банк / надходження

- **Етап «фундамент»**: усе вручну — бухгалтер вносить факт надходження, прив'язує до замовлення.
- **Етап «інтеграція»**: API клієнт-банків:
  - **Райфайзен** (основний рахунок ТОВ)
  - **Mono / Універсал Банк**
  - Імпорт виписки **CSV** (як проміжний/резервний варіант).
- Інтеграція дає «банківський фід»; зіставлення транзакції з замовленням бухгалтер усе одно підтверджує вручну (прибрати помилки ручного вводу сум, але не автозіставлення наосліп).

## 6. Модель даних (схема `tosho`, попередня)

> Назви колонок — англ., бізнес-сенс — укр. Орієнтовно; фіналізуємо при реалізації фази 1.

```
finance_legal_entities          -- НАШІ юрособи (ТОВ, ФОП1, ФОП2, +псевдо "individual")
  id, name, kind(llc|sole_prop|individual), vat_payer(bool),
  tax_group, edrpou, ipn, iban, requisites(jsonb), is_active

finance_accounts                -- каси/гаманці (контури)
  id, legal_entity_id, name, kind(bank|cash|crypto|personal_card),
  currency, bank_provider(raiffeisen|mono|manual|null),
  is_sensitive(bool),           -- true для фіз/готівка/крипта
  is_active

finance_invoices                -- рахунки клієнтам
  id, number, legal_entity_id, customer_id, order_id,
  issue_date, due_date, amount, vat_rate, vat_amount,
  prepayment_amount, balance_amount, status,
  file_pdf, file_xlsx, created_by

finance_payments                -- ФАКТ надходження грошей
  id, account_id, order_id(NOT NULL), invoice_id(nullable),
  amount, currency, fx_rate, uah_equivalent,
  paid_at, channel, source(manual|raiffeisen|mono|csv),
  bank_txn_ref, entered_by

finance_order_meta              -- фін-надбудова над замовленням
  order_id(PK), order_type(goods|services),
  payment_channel, legal_entity_id

finance_expense_categories      -- статті витрат
  id, name, kind(variable|fixed|tax|payroll), default_legal_entity_id

finance_expenses                -- витрати (гроші з гаманця)
  id, legal_entity_id, account_id, category_id, supplier_id,
  amount, vat_amount, expense_date,
  is_recurring(bool), recurrence(monthly|null),
  file, entered_by

finance_expense_allocations     -- розподіл 1 витрати на N замовлень
  id, expense_id, order_id, amount

finance_suppliers               -- постачальники (можливо reuse contractors — перевірити)
  id, name, requisites(jsonb), ...

finance_team_contracts          -- договори з командою (ФОП/ТОВ)
  id, member_user_id, legal_entity_id, kind(fop|llc),
  monthly_amount, start_date, end_date

finance_team_payouts            -- виплати команді (місячні)
  id, contract_id, member_user_id, period(month),
  amount, act_file, paid_at, status, account_id

finance_taxes                   -- податкові зобов'язання
  id, legal_entity_id, tax_type(vat|single_tax|esv|military),
  period, base_amount, rate, amount_due, due_date, paid_at, status

finance_documents               -- документи реалізації (видаткова/акт)
  id, type(goods_invoice|service_act), order_id, invoice_id,
  number, doc_date, counterparty_id,
  file_pdf, file_xlsx, edm_status(none|signed_external)

finance_reconciliations         -- акти звірки (снепшоти)
  id, counterparty_id, counterparty_kind(customer|supplier),
  legal_entity_id, period_from, period_to,
  snapshot(jsonb),              -- рахунки/оплати/баланс на момент генерації
  file_pdf, file_xlsx, generated_at, sent_at, sent_channel
```

**Ключове, що треба продумати на фазі 1:**
- Замовлення в системі — це не проста таблиця `orders`, а **похідні записи** (`src/features/orders/orderRecords.ts`). Фін-дані вішаємо окремою надбудовою `finance_order_meta`, ключ — `order_id`/`quote_id`. Не ламати наявну деривацію.
- Прив'язка оплати — `order_id NOT NULL`. Прорахунок (estimate) платити не може — на рівні UI і БД.

## 7. Ролі та доступи

Розширюємо наявну систему `module_access.finance` (уже заведено) гранулярністю:

| Роль | Маржа/собівартість | Сірі канали (фіз/готівка/крипта) | Обсяг |
|---|---|---|---|
| CEO (owner) | ✅ | ✅ | усе |
| Головбух (chief_accountant) | ✅ | ✅ | усе |
| Бухгалтер (accountant) | ❌ | ❌ | оплати + документи, без маржі/собівартості |
| Менеджер (manager) | ❌ | ❌ | тільки свої замовлення (оплати по них) |

Технічно: окрім `module_access.finance` (вхід у розділ) потрібні під-рівні:
- `finance_can_see_margin` (CEO + chief_accountant)
- `finance_can_see_sensitive_channels` (CEO + chief_accountant)
- менеджер — фільтрація по `manager_user_id` його замовлень (як уже зроблено в інших розділах).

Ролі `chief_accountant` / `accountant` / `manager` уже існують у `JOB_ROLE_LABELS` (`src/pages/TeamMembersPage.tsx`).

## 8. Етапність (щоб не робити все одним заходом)

Це ERP-lite, не «модуль на тиждень». Пропоную фази; кожна — самодостатня цінність.

**Фаза 1 — Контури + продажі + базовий дашборд** *(дає «скільки заробили / хто винен / по якому каналу»)*
- Юрособи + каси/гаманці (Налаштування → Юрособи)
- Фін-надбудова над замовленням (тип, канал, юрособа)
- Рахунки клієнтам (ручне створення, статуси)
- Оплати з прив'язкою до замовлення (ручний ввід)
- Дашборд: заробили / дебіторка / баланси кас по контурах

**Фаза 2 — Витрати + маржа**
- Статті витрат, сталі + змінні витрати
- Алокація витрати на кілька замовлень
- Маржа план/факт по замовленню
- Постачальники

**Фаза 3 — Документи**
- Шаблони + генерація рахунків/видаткових/актів (PDF+Excel)
- Список документів з фільтром
- `edm_status` для зовнішнього підпису

**Фаза 4 — Податки + календар + зарплати**
- Податки (ПДВ; ФОП ЄП/ЄСВ/ВЗ)
- Виплати команді + договори
- Платіжний календар (вихідні платежі + податки + зарплати)

**Фаза 5 — Банк**
- Райфайзен + Mono API, CSV-імпорт, банківський фід + ручне підтвердження зіставлення

**Фаза 6 — Звірки + звіти**
- Генерація актів звірки + архів снепшотів + надсилання
- Звіти (продажі/дебіторка/кредиторка/прибуток/ПДВ/розбивки)

## 9. Припущення, які треба підтвердити

1. **Крипта**: фіксуємо суму в крипті + UAH-еквівалент за курсом на момент надходження (курс вносить бухгалтер вручну). Підтвердити.
2. **«Реєстр продажів»** як назва (замість «Платіжний реєстр», щоб не плутати з «Платіжним календарем»). Підтвердити.
3. **Постачальники**: окрема таблиця чи reuse наявних «Підрядників» (`contractors`)? Перевірити при фазі 2.
4. **Звірки — архів снепшотів**: тримаємо історію згенерованих звірок (бо через місяць цифри інші). Прийнято як так; підтвердити.
5. **«Перші 15 днів» / вхідні залишки**: ручна форма введення стартових оплат/витрат станом на 01.06.2026. Уточнити формат.

## 10. Що НЕ входить (поки)

- Автоматичне податкове декларування / подача звітності в податкову
- Повна подвійна бухгалтерія (дебет/кредит проводки) — ведемо управлінський облік, не бухгалтерський регістр
- Зарплатні нарахування з усіма утриманнями (тільки факт виплат за договорами)
- Мультивалютність крім крипти (основна валюта — UAH)
